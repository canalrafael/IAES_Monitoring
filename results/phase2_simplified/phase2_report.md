## 1. Model Overview

Phase 2 trains a lightweight MLP (Multi-Layer Perceptron) to detect cross-core
microarchitectural interference in real-time on a Raspberry Pi 4. The model is
intentionally tiny to satisfy an embedded 5 KB memory budget.

### 1.1 Feature Engineering

Twelve features are extracted from four ratio signals using a sliding window of W = 10 samples.
Raw hardware counters are never used directly — only their ratios, which are
**workload-normalised** and more invariant across different programs.

| Signal | Formula | Physical meaning |
|---|---|---|
| IPC | `INSTRUCTIONS / CPU_CYCLES` | Computational efficiency (stalls reduce this) |
| MPKI | `(CACHE_MISSES × 1000) / INSTRUCTIONS` | Cache miss intensity per instruction |
| L2_PRESSURE | `L2_CACHE_ACCESS / CPU_CYCLES` | Memory bus utilisation |
| BRANCH_MISS_RATE | `BRANCH_MISSES / INSTRUCTIONS` | Branch predictor perturbation |

Each signal generates **three rolling statistics** over W = 10 samples:

| Statistic | Formula | Detects |
|---|---|---|
| `_mean` | Rolling average over W | Sustained level changes (e.g., baseline IPC drop) |
| `_std` | Rolling standard deviation over W | Increased variability caused by interference |
| `_delta` | `signal[t] − signal[t − W]` | Onset: sudden change in signal level |

**Total: 4 signals × 3 statistics = 12 features.**

### 1.2 Architecture

```
Input(12) → Linear(16) → ReLU → Dropout(0.1)
          → Linear(16) → ReLU → Dropout(0.1)
          → Linear(1)  → Sigmoid
```

| Property | Value |
|---|---|
| Parameters | 497 |
| Memory footprint | ~2 KB (weights only) |
| Inference latency | < 1 µs (CPU-only) |
| Loss function | `BCEWithLogitsLoss(pos_weight = 2 × class_ratio)` |
| Optimiser | Adam (LR = 2e-3, weight_decay = 1e-4) |
| LR schedule | `ReduceLROnPlateau(factor=0.5, patience=5, min_lr=1e-5)` |
| Early stopping | Patience = 10 on validation loss |

**pos_weight** upweights attack samples by `2 × class_ratio` to compensate for class
imbalance; without it, the model would tend to predict benign for all samples
(the majority class at test time in deployment).

### 1.3 Temperature Calibration

After training, a scalar temperature `T > 0` is fitted on the calibration set by
minimising the negative log-likelihood of calibrated probabilities:

```
p_calibrated = sigmoid(logit / T)
```

- `T > 1` softens overconfident predictions (most common after training with pos_weight)
- `T < 1` sharpens confident predictions (rare)

This step does **not** change the ranking of outputs — it only rescales probabilities
to be better-calibrated for threshold selection.

### 1.4 Data Split

| Split | Benign files | Attack files | Purpose |
|---|---|---|---|
| Train (70%) | 7 | 9 | Model parameter fitting |
| Calibration (10% of train) | — | — | Temperature T, threshold τ selection |
| Test (30%) | 3–4 | 4 | **All metric reporting** |

The split is **stratified at the file level** (not the sample level) to prevent
data leakage: no sample from a test file is ever seen during training.

## 2. Training Curve Outputs

### `loss_curve.png` — BCE Loss Curve

**What it shows:** Training and validation BCE (Binary Cross-Entropy) loss
over epochs, with an EMA-smoothed overlay for readability.

**How to read it:**
- The **training loss** (blue) should decrease monotonically as the model fits the data.
- The **validation loss** (coral) should decrease initially and then plateau or rise
  slightly (indicating overfitting onset).
- The **gap** between train and val loss indicates generalisation: a large, growing gap
  means the model has memorised training patterns.
- **Early stopping** (dashed red vertical line) fires when val loss has not improved
  by ≥ 1e-4 for 10 consecutive epochs, preserving the best-performing checkpoint.
- **LR schedule** drops appear as discontinuities in the loss slope — after a plateau,
  Adam halves the learning rate, allowing finer convergence.

**Key diagnosis:**
- If early stop fires before epoch 30, the model under-trains → increase patience or LR.
- If val loss is still decreasing at early stop → increase patience.
- If train loss → 0 but val loss plateaus high → overfitting → increase dropout or reduce units.

---

### `learning_curve.png` — Accuracy Curve

**What it shows:** Training and validation **classification accuracy** (%) over epochs.

> **Note:** Accuracy alone is an unreliable metric for imbalanced classes.
> A model that always predicts "benign" achieves ~55% accuracy on typical splits.
> These curves are provided for diagnostic purposes alongside loss and ROC curves.

**How to read it:**
- Both curves should rise and plateau close together for a well-generalising model.
- A large **train–val accuracy gap** (> 5%) indicates overfitting.
- The final val accuracy is a rough proxy for balanced F1; the ROC curve provides
  the definitive performance characterisation.

## 3. ROC / Pareto Evaluation Outputs

### Core Evaluation Methodology

Unlike classical train-time loss, the real-world deployment constraint is:

> **Recall ≥ 0.99** (miss fewer than 1% of attacks)  
> **FPR ≤ 0.10** (fewer than 10% of benign samples flagged as attacks)

A single fixed threshold τ cannot capture this trade-off across all workloads.
Instead, **all metrics are swept over τ ∈ [0, 1]** to generate a full ROC curve
for each smoothing window N.

---

### `roc_pareto.png` — Full ROC / Pareto Curve

**What it shows:** True Positive Rate (Recall) vs. False Positive Rate across all
threshold values τ, for each temporal smoothing level N ∈ {1, 3, 5, 7}.

| Axis | Metric | Formula |
|---|---|---|
| X-axis | False Positive Rate (FPR) | `FP / (FP + TN)` |
| Y-axis | Recall (True Positive Rate) | `TP / (TP + FN)` |

**How to read it:**
- A curve in the **top-left corner** is better (high recall, low FPR simultaneously).
- The **Area Under the Curve (AUC)** summarises the model's capability across all
  operating points; a perfect model has AUC = 1.0, a random model has AUC = 0.5.
- The **red shaded rectangle** marks the target operating region (Recall ≥ 0.99, FPR ≤ 0.10).
- **★ markers** show the best operating point within or nearest to the target region.
- The **dashed grey line** is the Logistic Regression baseline, evaluated identically.

**Smoothing effect (N):** Temporal smoothing computes a causal rolling mean of the
model's output probabilities over N consecutive samples before applying the threshold.
A larger N filters out transient false positives at the cost of N-sample detection delay.

---

### `roc_zoom_operational.png` — Zoomed Operational Region (FPR ≤ 0.40)

**What it shows:** Same as `roc_pareto.png` but zoomed into FPR ≤ 0.40 and
Recall ∈ [0.85, 1.0] to reveal fine differences between smoothing levels.

**Why it matters:** The full ROC curve compresses the interesting operating region
into a small corner. This zoom makes differences between N=1 and N=7 visible,
and shows how close the model comes to the target rectangle.

---

### `roc_pareto_analysis.png` — Pareto Summary Bar Chart

**What it shows:** For each smoothing window N, the best achievable FPR subject to
Recall ≥ 0.99 is shown as a bar, with reference lines from earlier model versions.

**How to read it:**
- Lower bars are better (lower FPR at the recall constraint).
- Reference lines show V2 N=1 and V2 N=7 performance, allowing direct comparison.
- If a bar reaches or passes below the red target line (FPR = 0.10), the model
  meets the deployment requirement for that smoothing level.

---

### `roc_pareto_validation.png` — Validation-Set Pareto Curve

**What it shows:** Same ROC sweep computed on the **validation set** (10% of train data)
rather than the held-out test set, for comparison.

**Why it matters:** Comparing validation-set vs. test-set ROC curves reveals
**distribution shift**: if the validation ROC is significantly better than the test ROC,
the threshold τ selected on validation data does not transfer to deployment.

---

### `threshold_sensitivity.png` — Threshold Sensitivity Analysis

**What it shows:** Recall (blue, left axis) and FPR (coral, right axis) plotted as
functions of threshold τ ∈ [0, 1] for the best-performing smoothing window.

**How to read it:**
- The **window where both Recall ≥ 0.99 and FPR ≤ 0.10 simultaneously** (if it exists)
  is the valid operating range for τ.
- A **wide valid window** means the model is robust to threshold drift — safe for deployment.
- A **narrow window** (< 0.05 in τ width) means performance is fragile; small miscalibration
  will violate one constraint.
- The reference horizontal lines mark the target thresholds (Recall = 0.99 in blue,
  FPR = 0.10 in coral).

> **Critical deployment note:** The τ selected on a calibration set does not
> automatically transfer to new workloads. This plot quantifies the sensitivity.
> In production, τ should be recalibrated or conservatively set lower.

## 4. Confusion Matrices

### `cm_N{1,3,5,7}.png` — Confusion Matrices per Smoothing Window

Four confusion matrices, one per smoothing level N ∈ {1, 3, 5, 7}, computed at
the best-τ operating point for that N (selected to maximise F1 given Recall ≥ 0.99).

**Cell definitions:**

|  | Predicted Benign | Predicted Attack |
|---|---|---|
| **True Benign** | TN — correct benign classification | FP — false alarm |
| **True Attack** | FN — missed attack (critical!) | TP — correct detection |

**Key cells:**
- **FN (False Negatives)** must be minimised — these are missed attacks. The model
  is trained with `pos_weight` to make FN rare at the cost of more FP.
- **FP (False Positives)** trigger unnecessary intervention. High FP rates cause
  alert fatigue and degrade system availability.

**Effect of increasing N:** Larger N tends to reduce FP (smoothing filters transient
spikes) while marginally increasing FN (delayed detection of brief attacks).

---

### `confusion_matrix_final.png` — Best Overall Confusion Matrix

The confusion matrix for the N value with the best FPR − Recall trade-off
(highest recall − FPR difference), presented at publication quality with
percentage annotations.

## 5. Probability Distribution

### `prob_distribution.png` — Output Probability Distribution

**What it shows:** Kernel density estimate (KDE) histograms of the model's
predicted probability `p(attack)` for benign samples (blue) and attack samples
(coral), on the held-out test set.

**How to read it:**
- Ideal case: **two sharp, non-overlapping peaks** — benign near 0.0, attack near 1.0.
- **Overlap region** (where both distributions have density) is where the threshold
  τ must operate. All samples in this overlap are ambiguous and will be misclassified
  by some τ.
- A bimodal structure in the benign distribution indicates that some benign workloads
  are being consistently confused with attacks (workload coverage gap).

**Interpretation:**
- If the attack peak is near 1.0 → the model is very confident on attacks ✓
- If the benign peak is near 0.0 → the model correctly suppresses benign alerts ✓
- If the benign distribution has a long tail toward 1.0 → FPR floor from specific
  workload types not represented in training (requires more diverse training data)

## 8. Logistic Regression Baseline

All ROC plots include a **Logistic Regression (LR) baseline**, trained on the same
features and evaluated with the same ROC protocol.

**Purpose of the baseline:**
- If **MLP AUC ≈ LR AUC**: the feature space is linearly separable; model complexity
  beyond LR adds no value. The bottleneck is the feature design, not model capacity.
- If **MLP AUC >> LR AUC**: the decision boundary is non-linear; the MLP is learning
  useful higher-order interactions between features.
- If **LR AUC < 0.5**: the features in that configuration are not linearly separable
  in the current representation (seen in V2 z-score features where LR AUC = 0.46).

**In the current results (phase2_simplified):** LR AUC ≈ 0.948, MLP AUC ≈ 0.978 —
indicating mild non-linearity that the MLP exploits, but the feature space is close
to linearly separable.

## 10. Key Definitions Reference

| Metric | Full Name | Formula | Good value |
|---|---|---|---|
| **Recall** | True Positive Rate | `TP / (TP + FN)` | ≥ 0.99 (target) |
| **FPR** | False Positive Rate | `FP / (FP + TN)` | ≤ 0.10 (target) |
| **Precision** | Positive Predictive Value | `TP / (TP + FP)` | Higher is better |
| **F1** | Harmonic mean of Prec & Recall | `2·P·R / (P + R)` | Higher is better |
| **AUC** | Area Under ROC Curve | ∫ TPR d(FPR) | Closer to 1.0 |
| **Cohen's d** | Standardised mean difference | `(µ₀ − µ₂) / σ_pooled` | \|d\| ≥ 0.8 = strong |
| **τ (tau)** | Decision threshold | P(attack) > τ → alert | Dataset-dependent |
| **N** | Smoothing window | Rolling mean of last N probs | Trade-off: latency vs FPR |
| **T** | Temperature | Calibration scalar for logits | Fitted post-training |
| **AUC (ROC)** | Discrimination ability | Probability correct ranking | 0.98 = excellent |
